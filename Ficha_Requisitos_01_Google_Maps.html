1. Mapeie os Atores

Quem usa o sistema? Pense em: Usuário final, Administrador, Fornecedor/Vendedor, Prestador de serviço, Suporte técnico.

Tipo de Ator Descrição resumida Objetivo principal no sistema
Ator 1 (Principal) Usuário comum, pessoa que busca endereços, rotas e informações de locais. Representa a maior base: 2 bilhões de usuários mensais ativos (Alphabet Q3 2024). Encontrar um endereço e obter a melhor rota até o destino de forma rápida e precisa.

Ator 3 (Negócio) Comerciante, proprietário de estabelecimento que cadastra seu negócio via Google Meu Negócio. São mais de 250 milhões de locais cadastrados na plataforma. Tornar seu estabelecimento visível no mapa para atrair clientes, exibindo horário, telefone e fotos.
Ator 4 (Integrador) Desenvolvedor, profissional que utiliza a Google Maps Platform API. Mais de 1,2 milhão de empresas e 5 milhões de apps/sites integram a API. Incorporar funcionalidades de mapa (rotas, geocodificação) em sistemas externos via API, sujeito ao SLA de 99,9%.

4. Requisitos Não Funcionais

Baseados no modelo de qualidade ISO 25010: desempenho, segurança, disponibilidade, usabilidade e escalabilidade.

Desempenho
Requisito:
O mapa deve carregar em até 3 segundos em conexão 4G. As sugestões de busca devem aparecer em até 500 milissegundos. A API de Geocoding responde em ~120ms em condições normais.
Como medir / Métrica:
Tempo de carregamento < 3s (4G); latência de autocomplete < 500ms. O próprio SLA do Google define "downtime" como resposta acima de 3 segundos por 10 minutos consecutivos.
Segurança
Requisito:
A localização do usuário deve ser coletada somente com consentimento explícito (conforme LGPD/GDPR). Toda transmissão de dados deve usar HTTPS com TLS 1.2+ (o Google suporta TLS 1.2 e 1.3).
Como medir / Métrica:
100% das conexões via HTTPS; zero coletas de localização sem permissão explícita do usuário.
Disponibilidade
Requisito:
O sistema deve manter 99,9% de uptime mensal, conforme o SLA oficial da Google Maps Platform. Isso equivale a no máximo ~43 minutos de indisponibilidade por mês.
Como medir / Métrica:
Uptime ≥ 99,9% mensal; monitoramento via Google Maps Platform Status Dashboard. Créditos financeiros de 10% a 50% são emitidos quando o SLA é descumprido.
Usabilidade
Requisito:
O aplicativo deve ser operável com uma mão só (uso em movimento). Deve oferecer suporte a leitor de tela, navegação por teclado e voz. O banco de dados contém 50+ milhões de locais com informações de acessibilidade (cadeirante, banheiro adaptado).
Como medir / Métrica:
Tamanho mínimo de botões: 48×48px. O Google não possui certificação WCAG formal para o Maps, e o mapa interativo é reconhecidamente difícil para leitores de tela.
Escalabilidade
Requisito:
O sistema deve suportar 2 bilhões de usuários mensais ativos (~50 sessões/mês por usuário) e processar mais de 20 milhões de contribuições diárias (avaliações, fotos, edições) sem degradação perceptível.
Como medir / Métrica:
Infraestrutura global do Google Cloud com auto-scaling; 13 APIs cobertas pelo SLA; dashboard de métricas por projeto (Traffic, Errors, Median Latency).

Fontes: Google Maps Platform SLA (cloud.google.com/maps-platform/terms/sla) | ISO 25010 Software Quality Model | Google GAAD 2024 Accessibility Update

5. Cenário de Exceção

Escolha uma situação crítica e descreva o comportamento esperado do sistema. Cenário baseado em comportamento real documentado.

O que aconteceu?
O serviço de dados de trânsito em tempo real ficou indisponível, impedindo o cálculo de rotas otimizadas. Incidente real de referência: em fevereiro de 2026, as APIs de Geocoding e Places ficaram instáveis por ~36 horas na América do Norte (Incidente 163VbP2njo8deNYScb4v).
Quem foi afetado?
Usuário (motorista em navegação ativa): recebe rota sem considerar o trânsito, podendo pegar caminhos congestionados. Desenvolvedores: APIs retornam erros 5xx ou latência elevada (P90 e P99).
Qual a ação esperada do sistema?
A Routes API alterna automaticamente para o modo TRAFFIC_UNAWARE, que calcula rotas usando dados históricos/estáticos de distância. No app mobile, ativa mapas off-line baixados previamente. O sistema aplica degradação graciosa (graceful degradation): perde precisão de trânsito, mas mantém a navegação funcional.
Qual mensagem o usuário deve ver?
"Informações de trânsito temporariamente indisponíveis. A rota sugerida pode não refletir as condições atuais."
O que acontece com os dados?
A rota é calculada normalmente usando dados estáticos. O sistema utiliza dados em cache da última atualização disponível e tenta reconectar ao serviço periodicamente de forma automática. A camada de trânsito no mapa simplesmente desaparece ou mostra o último estado conhecido. A média histórica é de ~1,3 incidentes por mês na plataforma (dados do Status Dashboard, últimos 5 anos: 80+ incidentes documentados).

Fontes: Google Maps Platform Status Dashboard | Routes API Documentation (config_trade_offs) | Incidente 163VbP2njo8deNYScb4v

7. Observações Finais / Aprendizados

Análise técnica dos aprendizados obtidos com este exercício.

A análise do Google Maps como exercício de engenharia de requisitos evidencia a diferença entre a perspectiva de usuário final e a perspectiva de analista. Do ponto de vista do usuário, o Maps é um app de rotas. Do ponto de vista técnico, trata-se de uma plataforma com múltiplas camadas: busca geoespacial, renderização cartográfica, processamento de rotas em tempo real, coleta colaborativa de dados (crowdsourcing com 150 milhões de contribuidores) e uma API aberta para integração com sistemas externos (1,2 milhão de empresas clientes). Na identificação dos atores, o caso do Contribuidor Local demonstra que nem todos os atores são óbvios. Esse ator não é cliente pagante nem funcionário, mas sem ele a base de dados do sistema perderia qualidade, pois são 20 milhões de contribuições diárias. Isso reforça que o mapeamento de atores exige investigar toda a cadeia de valor: quem alimenta os dados, quem consome, quem mantém e quem administra. Requisitos levantados a partir de um único perfil de ator tendem a ser incompletos. A priorização MoSCoW expôs uma armadilha recorrente: confundir "funcionalidade que agrada o usuário" com "funcionalidade que sustenta a proposta de valor do negócio". O trânsito em tempo real (#9) foi reclassificado de Must para Should após verificar que o Maps existiu por anos sem esse recurso. Segundo o DSDM (criador do MoSCoW), Must Have deve consumir no máximo 60% do esforço total, e ultrapassar esse limite é indicador de risco no projeto. Os requisitos não funcionais demonstram que as restrições de qualidade são tão determinantes quanto as funcionalidades. O SLA oficial de 99,9% de uptime (máximo ~43 min/mês de queda) é um compromisso contratual com créditos financeiros em caso de descumprimento, e não é um número arbitrário. Isso indica que RNFs devem ser derivados do cenário real de operação, com métricas mensuráveis e justificáveis, conforme preconiza a norma ISO 25010 e o padrão IEEE 830. O cenário de exceção foi a seção de maior valor analítico. O Maps implementa degradação graciosa (graceful degradation): quando os dados de trânsito ficam indisponíveis, a Routes API alterna para o modo TRAFFIC_UNAWARE, usando dados históricos. O sistema perde precisão, mas não interrompe a operação. O incidente real de fevereiro de 2026 (~36 horas de instabilidade) demonstra que esse cenário não é teórico, pois ocorre na prática com frequência média de 1,3 vezes por mês. Para a fase de levantamento de requisitos do projeto Mapa de Sala, isso implica que as perguntas aos stakeholders não devem se limitar a "o que o sistema deve fazer", mas incluir "o que acontece hoje quando o processo falha", porque o procedimento manual atual é o ponto de partida para o desenho da contingência no sistema.
