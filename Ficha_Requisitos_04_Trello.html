Roteiro de Análise de Requisitos

Análise do sistema Trello, ferramenta de gestão visual de projetos baseada em Kanban | 50+ milhões de usuários registrados

Sistema analisado: Trello
Data: 25/06/2026
Analista: Thaís Oliveira
1. Mapeie os Atores

O Trello é um sistema colaborativo: múltiplos atores operam simultaneamente sobre o mesmo recurso (o quadro). Isso cria desafios de concorrência distintos dos sistemas anteriores.

Tipo de Ator Descrição resumida Objetivo principal no sistema
Ator 1 (Principal) Membro do Quadro: usuário que cria, move e edita cartões dentro de um quadro. A plataforma tem 50+ milhões de usuários registrados. No plano gratuito, quadros são limitados a 10 colaboradores. Organizar tarefas visualmente em colunas (listas), movendo cartões entre etapas do fluxo de trabalho.
Ator 2 (Administrador) Administrador do Workspace: gerencia membros, permissões e configurações do workspace. Existem 3 níveis de papel no workspace (admin, normal, guest) e 3 no quadro (admin, normal, observer). Controlar acesso, gerenciar licenças (Free, Standard, Premium, Enterprise) e definir políticas de visibilidade dos quadros.
Ator 3 (Sistema) Power-Ups / Automações (Butler): integrações e automações que executam ações automaticamente. Butler permite regras, botões e comandos de calendário. Power-Ups conectam Slack, GitHub, Google Drive, etc. Automatizar ações repetitivas (ex.: mover cartão quando checklist completa, notificar no Slack quando prazo vence) e integrar dados externos.
Ator 4 (Observador) Observer: papel com permissão somente leitura, disponível nos planos Premium e Enterprise. Pode visualizar quadros mas não editar cartões, listas ou configurações. Acompanhar progresso do projeto sem risco de alteração acidental. Útil para stakeholders e gestores que precisam de visibilidade sem participação operacional.

2. Liste 10 Funcionalidades Principais

O modelo mental do Trello é o Kanban: quadro → listas (colunas) → cartões (itens de trabalho). Todas as funcionalidades derivam dessa hierarquia.

# Funcionalidade Ator que executa
1Criar quadro com listas (colunas) personalizáveis representando etapas do fluxo de trabalhoMembro / Admin
2Criar, editar e excluir cartões dentro de listas, com título, descrição (Markdown), membros atribuídos, etiquetas coloridas e data de entregaMembro
3Arrastar e soltar cartões entre listas (drag-and-drop), sincronizado em tempo real via WebSocket (Socket.io modificado) para todos os membros do quadroMembro
4Adicionar checklists com itens marcáveis e barra de progresso percentual dentro do cartãoMembro
5Anexar arquivos ao cartão (upload direto, Google Drive, Dropbox, OneDrive). Limite de 10MB/arquivo no plano Free, 250MB nos planos pagosMembro
6Comentar em cartões com menções (@membro) que geram notificação por e-mail e pushMembro
7Filtrar cartões por etiqueta, membro atribuído, data de entrega ou texto no título/descriçãoMembro
8Configurar automações via Butler: regras (quando X acontecer, faça Y), botões de quadro/cartão e comandos agendados por calendárioMembro / Admin
9Gerenciar permissões de membros no quadro (admin, normal, observer) e visibilidade (privado, workspace, público)Admin do Workspace
10Visualizar atividade do quadro em feed cronológico (Activity Log) com histórico completo de açõesMembro / Observer

4. Requisitos Não Funcionais

Baseados no modelo ISO 25010 e na stack tecnológica real do Trello.

Desempenho
Requisito:
A sincronização de drag-and-drop deve refletir em todos os clientes conectados em <500ms. O Trello usa WebSockets via Socket.io modificado, com fallback para HTTP long-polling em redes restritivas. O sistema impõe limite de 5.000 cartões abertos por quadro (hard limit), e a performance degrada perceptivelmente a partir de ~1.000 cartões por lista.
Como medir / Métrica:
Latência de sync P95 < 500ms; tempo de abertura de quadro com 500 cartões < 3s; limite documentado de 5.000 cartões/quadro.
Segurança
Requisito:
Dados em trânsito via TLS 1.2+. Dados em repouso criptografados (AES-256). Autenticação via Atlassian Account com suporte a SSO (SAML 2.0) nos planos Premium/Enterprise. 2FA disponível para todos os planos. Conformidade com SOC 2 Type II e ISO 27001 (herdada da Atlassian Cloud).
Como medir / Métrica:
Certificações SOC 2 e ISO 27001 ativas; zero dados de cartão em texto puro em logs; sessões expiram após inatividade configurável.
Disponibilidade
Requisito:
A Atlassian Cloud (que hospeda o Trello) publica SLA de 99,9% de uptime para planos Premium e Enterprise. O Trello Status Page (status.trello.com, via Statuspage.io) registra incidentes com transparência, e interrupções maiores são tipicamente resolvidas em <4h. A stack roda em AWS com Node.js, MongoDB (banco principal), Redis (cache/sessões) e HAProxy (balanceamento).
Como medir / Métrica:
Uptime ≥ 99,9% mensal (≤ 43 min downtime/mês); MTTR < 4h para incidentes críticos.
Usabilidade / Acessibilidade
Requisito:
Interface responsiva (Mobile First). Modo de acessibilidade para daltonismo disponível desde 2014. Esse modo adiciona padrões de textura às etiquetas coloridas para distinguibilidade sem depender exclusivamente de cor (WCAG 2.1 nível AA). Suporte a navegação por teclado para todas as operações principais.
Como medir / Métrica:
WCAG 2.1 AA em todas as páginas; modo colorblind ativável em configurações; todas as ações acessíveis via teclado (Tab, Enter, Esc).
Escalabilidade
Requisito:
O sistema deve suportar 50+ milhões de usuários registrados com múltiplos quadros simultâneos. Stack: Node.js (servidor), MongoDB (banco NoSQL, escolha natural para o modelo flexível de cartões com campos variáveis), Redis (cache e gerenciamento de sessões WebSocket), HAProxy (balanceamento de carga). A escolha de MongoDB (documento) em vez de PostgreSQL (relacional) reflete o modelo de dados do Trello: cartões têm campos variáveis (checklists, anexos, campos customizados) que mapeariam em múltiplas tabelas de junção no modelo relacional.
Como medir / Métrica:
Horizontal scaling via sharding MongoDB; Redis Cluster para distribuição de sessões WebSocket; auto-scaling de instâncias Node.js na AWS.

5. Cenário de Exceção

Cenário baseado na estratégia real de resolução de conflitos do Trello.

O que aconteceu?
Dois membros editam o mesmo cartão simultaneamente: um altera o título enquanto o outro modifica a descrição. Em paralelo, um terceiro membro arrasta o cartão para outra lista. As três operações chegam ao servidor em ordem diferente da que foram executadas nos clientes.
Quem foi afetado?
Todos os membros do quadro, que podem ver estados inconsistentes por frações de segundo até a sincronização convergir. O membro cuja edição chegou por último tem sua versão preservada (last-write-wins).
Qual a ação esperada do sistema?
O Trello usa estratégia server-authoritative last-write-wins: o servidor é a fonte de verdade e a última operação recebida prevalece. O Trello NÃO usa CRDTs (Conflict-free Replicated Data Types) nem Operational Transformation (OT). Essas técnicas são usadas por editores de texto colaborativo (Google Docs), mas o modelo de dados do Trello (campos discretos: título, descrição, posição) torna last-write-wins suficiente. Cada campo é atualizado independentemente, e editar o título não conflita com editar a descrição do mesmo cartão. Para drag-and-drop, o servidor recalcula a posição ordinal do cartão na lista e propaga via WebSocket.
Qual mensagem o usuário deve ver?
O Trello não exibe mensagem explícita de conflito. A interface simplesmente atualiza para o estado mais recente via WebSocket. Se a conexão WebSocket cair, o sistema faz fallback para HTTP long-polling e exibe indicador de reconexão ("Reconectando..."). Ao reconectar, o cliente solicita o estado completo do quadro e atualiza a interface.
O que acontece com os dados?
Todas as operações são registradas no Activity Log com timestamp e autor, permitindo auditoria completa. Não há "undo" global, e o membro que perdeu sua edição precisa refazê-la manualmente. Para operações destrutivas (excluir cartão, arquivar lista), o Trello usa soft-delete: o item vai para o arquivo e pode ser restaurado. A escolha de last-write-wins é uma decisão de trade-off consciente: simplicidade de implementação e previsibilidade de comportamento vs. risco de perda silenciosa de edições. Para o volume típico de edição do Trello (dezenas de operações/minuto por quadro, não milhares), esse trade-off é adequado.

6. Pós-Análise (Validação com o app real)
Todos os atores estão corretos? Sim. Observer como ator distinto validado, papel exclusivo Premium/Enterprise.
As 10 funcionalidades existem no sistema? Todas verificadas. Limites de 5.000 cartões e 10MB/arquivo confirmados na documentação oficial.
A priorização MoSCoW bate com a realidade? Butler como Could Have validado, adquirido e integrado apenas em 2018, anos após o lançamento.
Os requisitos não funcionais são viáveis? Stack Node.js + MongoDB + Redis confirmada. SLA 99,9% da Atlassian Cloud verificado.
O cenário de exceção é tratado pelo sistema? Last-write-wins confirmado, sem CRDTs/OT. Fallback HTTP long-polling verificado na prática.
7. Observações Finais / Aprendizados
A análise do Trello encerra o ciclo das quatro fichas introduzindo o conceito de colaboração em tempo real sobre dados compartilhados, um problema de natureza diferente dos anteriores. No Google Maps, os dados são predominantemente lidos; no Airbnb, as escritas são transações discretas (reservar/cancelar); no Uber, o matching é efêmero e descartável. No Trello, múltiplos usuários editam o mesmo recurso persistente continuamente, o que coloca o problema de concorrência no centro da arquitetura. A estratégia de resolução de conflitos do Trello, server-authoritative last-write-wins, é deliberadamente simples. O sistema NÃO usa CRDTs (Conflict-free Replicated Data Types) nem Operational Transformation (OT), técnicas empregadas por editores de texto colaborativo como Google Docs. A razão é que o modelo de dados do Trello opera em campos discretos (título, descrição, posição, membros), não em texto contínuo. Quando dois usuários editam campos diferentes do mesmo cartão, não há conflito real, pois cada campo é atualizado independentemente. Quando editam o mesmo campo, o último a chegar ao servidor prevalece. Essa é uma decisão de trade-off documentável: simplicidade e previsibilidade vs. risco de perda silenciosa de edições. Para o volume típico do Trello (dezenas de operações por minuto por quadro), o trade-off é adequado. Para um editor de texto com dezenas de keystrokes por segundo, seria inaceitável. O mapeamento de atores revelou o papel de Observer, um ator com permissão exclusivamente de leitura, disponível apenas nos planos Premium e Enterprise. A existência desse papel reflete uma necessidade real de organizações: stakeholders e gestores precisam de visibilidade sobre o progresso do projeto sem risco de alteração acidental. Em termos de requisitos, isso significa que o sistema de permissões não é binário (acesso/sem acesso), mas granular, e a granularidade cresce com o tier do plano, o que cria uma interseção entre requisitos funcionais e modelo de negócio. A escolha de MongoDB como banco de dados principal não é arbitrária. O modelo de dados do Trello, com cartões de campos variáveis (checklists com N itens, anexos de tipos diversos, campos customizados definidos pelo usuário), mapeia naturalmente em documentos JSON aninhados. Em um modelo relacional (PostgreSQL), o mesmo dado exigiria múltiplas tabelas de junção (cartão_checklist, checklist_item, cartão_anexo, cartão_campo_custom), com consultas N+1 e complexidade crescente. O MongoDB permite armazenar o cartão inteiro como um documento único, o que simplifica leitura e escrita. O trade-off é que consultas analíticas complexas (ex.: "todos os cartões atrasados de todos os quadros de um workspace") são mais difíceis em documento do que em relacional. Para o projeto Mapa de Sala, onde o modelo de dados é mais estruturado (salas, turmas, horários com relações bem definidas), o modelo relacional é provavelmente mais adequado. O limite de 5.000 cartões abertos por quadro, com degradação perceptível a partir de ~1.000 por lista, é um exemplo de requisito não funcional que emerge da implementação. A interface carrega todos os cartões visíveis no DOM, sem virtualização de lista (técnica onde apenas os itens visíveis na viewport são renderizados). Isso significa que o limite não é uma decisão de produto, mas uma consequência da arquitetura de frontend. Na especificação de requisitos, limites operacionais como esse devem ser documentados explicitamente, para que os stakeholders não assumam capacidade ilimitada. A funcionalidade de acessibilidade para daltonismo, implementada desde 2014 com padrões de textura nas etiquetas, é um exemplo de requisito não funcional que vai além da conformidade legal (WCAG 2.1 AA). É uma decisão de design inclusivo que demonstra maturidade no tratamento de acessibilidade. Em vez de apenas garantir contraste mínimo, o sistema oferece um canal de informação redundante (cor + textura) que elimina a dependência exclusiva de percepção cromática.
